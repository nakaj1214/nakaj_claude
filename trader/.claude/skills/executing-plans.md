# 計画実行スキル

作成した計画を確実に実行し、各チェックポイントで検証するためのプロセス。

## 使う場面

1. writing-plansで計画を作成した後
2. 複数タスクを順番に実行するとき
3. 中断/再開が前提の長いタスク
4. チームが並行作業する場合

## 実行の原則

### 基本ルール

| ルール | 説明 |
|------|-------------|
| 一度に1つ | 複数タスクを同時に始めない |
| 完了確認 | 次へ進む前に検証する |
| 記録する | 進捗と結果を記録する |
| 問題は即報告 | ブロッカーを隠さない |

### 実行フロー

```
1. タスク開始を宣言
   -> 2. タスク実行
   -> 3. 完了条件を確認
   -> 4. チェックポイントで検証
   -> 5. 問題がなければ次へ
      （問題があれば修正または計画更新）
```


### AIツール選定

| タスク種別 | 推奨ツール |
|-----------|------------------|
| 複雑なロジック | Claude Code |
| UIプロトタイピング | Codex CLI |
| 大規模リファクタ | Gemini CLI |
| 定型の大量処理 | OpenCode + GLM |

## 実行コマンド例

### タスク開始

```bash
# 進捗ファイルを更新
echo "## 進行中 Task 1.1: API実装
- Started: $(date)
- Status: In progress" >> progress.md

# タスク実行
# ... 実装作業 ...

# 完了条件を確認
npm test
```

### チェックポイント実行

```bash
# 自動テスト
npm run test
npm run lint
npm run build

# 結果を記録
echo "### フェーズ1 チェックポイント
- Tests: $(npm test 2>&1 | tail -1)
- Build: Success
- Time: $(date)" >> progress.md
```

## 問題対応

### 判断フロー

```
問題発生
  -> [5分で解決できる?]
     -> Yes: 修正して続行
     -> No: ブロッカーとして報告
          -> [計画変更が必要?]
             -> Yes: 計画を更新して続行
             -> No: 支援を待つ
```

### ブロッカーレポートテンプレート

```markdown
## ブロッカーレポート

### タスク
Task 2.1: [タスク名]

### 問題
[問題の具体的な説明]

### 試した対応
1. [対応1] - 結果: [結果]
2. [対応2] - 結果: [結果]

### 必要なこと
- [ ] [必要なリソース/情報/判断]

### 影響
- 後続タスク: [影響を受けるタスク]
- 予定への影響: [想定される遅延]
```

```markdown
- ステータス: PASS
- フェーズ2開始: 10:50
```
