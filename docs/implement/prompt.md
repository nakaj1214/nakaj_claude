要約（日本語）

この文章は「Claude/Codex を最大限に活かせていない気がする」「他の人はロケット作ってるのに自分は石を積んでるだけに見える」という開発者のモヤモヤに対して、道具やハーネスの盛りすぎではなく、文脈（コンテキスト）設計と運用原則が本体だ、と言い切る内容です。結論は “少なく・明確に・分業し・終わりを定義し・ルールとスキルを育てつつ定期的に掃除する”。

1) 世界は高速で進化している。だから「足し算」より「引き算」

モデル世代が進むほど指示への従順さが増え、昔必要だった工夫が突然いらなくなる。

いま流行ってる外部ハーネスや依存を大量に入れると、将来のモデル進化で不要になった時に足かせになる。

本当に有用な仕組みは、結局フロンティア企業がプロダクトに取り込む（skills、memory、planning 等がそうだった）ので、過度に追いかけなくていい。

2) いちばん大事なのは「コンテキスト」。盛るほど壊れる

プラグイン・メモリ・巨大な CLAUDE.md で情報が多すぎると、タスクと無関係な過去の注意書きまで混ざって判断が鈍る（コンテキスト膨張）。

エージェントに渡す情報は「そのタスクに必要な最小限」に絞るべき。

3) 研究と実装は分ける。実装指示は具体的に

「認証を作って」みたいに曖昧だと、エージェントが選択肢調査まで始めて文脈が汚れ、実装段階で混乱・幻覚が増える。

先に調査（Research）で方針を確定し、新しいコンテキストで実装（Implementation）させるのが強い。

4) “迎合（sycophancy）”が幻覚を生む。中立プロンプトが効く

エージェントは「求められた成果」を出そうとして、バグを探せと言われれば“無理にでも”バグを捻り出すことがある。

「バグを探せ」より「ロジックを追って観察結果を報告せよ」など、結論を誘導しない中立指示が良い。

さらに迎合性を逆手に取り、
バグ列挙役（過剰に拾う）→反証役（潰す）→審判役（判定）
の三者で精度を上げるやり方を提案している。

5) エージェントは「始め方」は得意だが「終わらせ方」が苦手

人間は自然に完了条件を理解するが、エージェントは途中で止めたり、スタブだけ作って終わりがち。

だから テストやスクリーンショット検証など、終点を機械的に判定できる条件を置くと強い。

さらに {TASK}_CONTRACT.md のような「完了契約」を作り、契約が満たされるまで終了できない stop-hook を置く発想も紹介。

6) 長時間セッションで走らせ続けるのは最適とは限らない

24時間走らせると契約やタスクが混ざってコンテキストが太り、ドリフトしやすい。

代わりに 契約（コントラクト）ごとに新セッションで回す方が良い。

7) CLAUDE.md は「巨大な説明書」ではなく「条件分岐ディレクトリ」にせよ

CLAUDE.md 自体には全部を書かず、**状況に応じて読むべきルール/スキルへの案内（IF-ELSE）**を最小限に置く。

気に入らない挙動を見たら「ルール」として明文化し、それを読ませる。

手順・レシピ化できるものは「スキル」にして再現性を上げる。

8) ルールやスキルを増やすとまた性能が落ちる。だから定期的に「掃除」

増えるほど矛盾や重複でコンテキストが汚れる。

たまに統合・整理（矛盾解消、不要削除）する“スパ日”が必要。

9) 最後は自分が責任を持て

完璧なエージェントはいない。設計や実装を任せても、成果物の最終責任は人間が負う。
