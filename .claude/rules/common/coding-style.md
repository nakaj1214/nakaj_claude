# 共通コーディングスタイルルール

すべての言語とファイル種別に適用される。

## イミュータビリティ（最重要）

**常に新しいオブジェクトを作成し、既存のオブジェクトを変更しないこと。**

```typescript
// ❌ 悪い例 - 既存オブジェクトを変更している
user.name = newName;

// ✅ 良い例 - 新しいオブジェクトを作成している
return { ...user, name: newName };
```

隠れた副作用を防ぎ、安全な並行処理を可能にする。

## ファイル構成

- 1ファイルあたり **200〜400行** を目標とし、**800行** を上限とする
- 大きなモノリシックファイルよりも、焦点を絞った多数のファイルを優先する
- ファイル種別ではなく、**機能またはドメイン** で構造化する

## エラーハンドリング

**常にエラーを包括的に処理すること:**
- 各レベルで明示的に処理する
- ユーザーフレンドリーなUIメッセージ（内部情報を公開しない）
- サーバーサイドでの詳細なログ記録
- サイレント失敗は禁止（`catch (e) {}` は禁止）

## 入力バリデーション

**システム境界では常にバリデーションを行うこと:**
- すべてのユーザー入力
- 外部APIのレスポンス
- 処理前のファイル内容
- スキーマベースのバリデーション（Zod、Pydantic など）を使用

## コード品質チェックリスト

作業完了とマークする前に確認すること:
- [ ] 命名が明確で自己説明的である
- [ ] 関数が50行以内である
- [ ] ファイルが800行以内である
- [ ] ネストの深さが最大4レベルである
- [ ] すべてのエラーが処理されている
- [ ] ハードコードされた値がない（設定/定数を使用）
- [ ] イミュータブルパターンが全体で使用されている
