# テスト要件

## カバレッジ基準

すべてのモジュールで **最低80%のテストカバレッジ** を達成すること。

## 必須テスト種別

| 種別 | スコープ | 例 |
|------|-------|---------|
| **ユニット** | 個別の関数/クラス | `test_calculate_total_with_discount` |
| **インテグレーション** | システム間の連携 | `test_user_creation_persists_to_db` |
| **E2E** | ユーザーワークフロー全体 | `test_checkout_flow_completes_order` |

## テスト駆動開発サイクル

```
RED    → まず失敗するテストを書く
VERIFY → テストが実際に失敗することを確認する
GREEN  → テストを通す最小限のコードを実装する
VERIFY → テストが通ることを確認する
REFACTOR → クリーンアップし、グリーンを維持する
MEASURE → 80%以上のカバレッジを確認する
```

**常にテストを先に書くこと。例外なし。**

## TDD エージェント

新機能を実装する際は **tdd-guide** エージェントを使用して:
- 実装前にテストが書かれていることを確認する
- Red-Green-Refactor サイクルを維持する
- 完了後にカバレッジ指標を検証する

## テストの独立性

- 各テストは独立して実行される（共有されるミュータブルな状態がない）
- テストは任意の順序で実行可能
- テストは自分自身の後片付けをする
- ユニットテストでは外部サービスをモック化する

## テストが失敗した場合

1. テストを通すためにテストのアサーションを弱めない
2. テストではなく実際のコードを修正する
3. 行き詰まったら **tdd-guide** エージェントを使用する
