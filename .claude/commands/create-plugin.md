# プラグイン作成ワークフロー

初期コンセプトからテスト済みの実装まで、完全で高品質な Claude Code プラグインの作成をユーザーにガイドする。体系的なアプローチに従う: 要件の理解、コンポーネントの設計、詳細の明確化、ベストプラクティスに従った実装、検証、テスト。

## コア原則

- **明確化の質問をする**: プラグインの目的、トリガー、スコープ、コンポーネントに関するすべての曖昧さを特定する。前提を置くのではなく、具体的で明確な質問をする。実装に進む前にユーザーの回答を待つ。
- **関連スキルをロードする**: 必要に応じて Skill ツールを使って plugin-dev スキルをロードする（plugin-structure、hook-development、agent-development など）
- **専門エージェントを活用する**: AI 支援開発のために agent-creator、plugin-validator、skill-reviewer エージェントを活用する
- **ベストプラクティスに従う**: plugin-dev 自体の実装からパターンを適用する
- **段階的な情報開示**: リファレンス/サンプル付きの軽量スキルを作成する
- **TodoWrite を使用する**: すべてのフェーズで進捗を追跡する

**初期リクエスト:** $ARGUMENTS

---

## フェーズ 1: 調査

**目標**: どのようなプラグインを構築する必要があるか、どのような問題を解決するかを理解する

**アクション**:
1. 全7フェーズの todo リストを作成する
2. 引数からプラグインの目的が明確な場合:
   - 理解内容を要約する
   - プラグインの種類を特定する（統合、ワークフロー、分析、ツールキットなど）
3. プラグインの目的が不明確な場合、ユーザーに質問する:
   - このプラグインはどのような問題を解決するか？
   - 誰がいつ使うか？
   - 何をすべきか？
   - 参考になる類似プラグインはあるか？
4. 理解内容を要約し、進める前にユーザーに確認する

**出力**: プラグインの目的とターゲットユーザーの明確な記述

---

## フェーズ 2: コンポーネント計画

**目標**: 必要なプラグインコンポーネントを決定する

**このフェーズの前に plugin-structure スキルをロードすること**（Skill ツールを使用）。

**アクション**:
1. plugin-structure スキルをロードしてコンポーネントの種類を理解する
2. プラグインの要件を分析し、必要なコンポーネントを決定する:
   - **Skills**: 専門知識が必要か？（hooks API、MCP パターンなど）
   - **Commands**: ユーザー起動のアクションか？（デプロイ、設定、分析）
   - **Agents**: 自律的なタスクか？（検証、生成、分析）
   - **Hooks**: イベント駆動の自動化か？（検証、通知）
   - **MCP**: 外部サービス統合か？（データベース、API）
   - **Settings**: ユーザー設定か？（.local.md ファイル）
3. 必要なコンポーネントの種類ごとに以下を特定する:
   - 各種類の数
   - 各コンポーネントの役割
   - おおまかなトリガー/使用パターン
4. コンポーネント計画をテーブルでユーザーに提示する:
   ```
   | コンポーネント種類 | 数 | 目的 |
   |----------------|-------|---------|
   | Skills         | 2     | Hook パターン、MCP の使い方 |
   | Commands       | 3     | デプロイ、設定、検証 |
   | Agents         | 1     | 自律的な検証 |
   | Hooks          | 0     | 不要 |
   | MCP            | 1     | データベース統合 |
   ```
5. ユーザーの確認または調整を得る

**出力**: 作成するコンポーネントの確認済みリスト

---

## フェーズ 3: 詳細設計と明確化の質問

**目標**: 各コンポーネントを詳細に仕様化し、すべての曖昧さを解消する

**重要**: これは最も重要なフェーズの1つ。スキップしないこと。

**アクション**:
1. 計画内の各コンポーネントについて、仕様が不十分な側面を特定する:
   - **Skills**: 何がトリガーするか？どのような知識を提供するか？どの程度詳細か？
   - **Commands**: どのような引数か？どのツールか？インタラクティブか自動か？
   - **Agents**: いつトリガーするか（プロアクティブ/リアクティブ）？どのツールか？出力形式は？
   - **Hooks**: どのイベントか？プロンプトベースかコマンドベースか？検証基準は？
   - **MCP**: どのサーバータイプか？認証は？どのツールか？
   - **Settings**: どのフィールドか？必須 vs オプションか？デフォルト値は？

2. **すべての質問を整理されたセクションでユーザーに提示する**（コンポーネント種類ごとに1セクション）

3. **実装に進む前に回答を待つ**

4. ユーザーが「おまかせで」と言った場合、具体的な推奨事項を提示し明示的な確認を得る

**スキルに対する質問例**:
- どのようなユーザークエリがこのスキルをトリガーすべきか？
- ユーティリティスクリプトを含めるべきか？どのような機能か？
- コアの SKILL.md はどの程度詳細にすべきか vs references/ はどうか？
- 含めるべき実際のサンプルはあるか？

**エージェントに対する質問例**:
- このエージェントは特定のアクション後にプロアクティブにトリガーすべきか、それとも明示的に要求されたときのみか？
- どのツールが必要か（Read、Write、Bash など）？
- 出力形式はどうあるべきか？
- 適用すべき特定の品質基準はあるか？

**出力**: 各コンポーネントの詳細仕様

---

## フェーズ 4: プラグイン構造の作成

**目標**: プラグインのディレクトリ構造とマニフェストを作成する

**アクション**:
1. プラグイン名を決定する（kebab-case、説明的）
2. プラグインの配置場所を選択する:
   - ユーザーに尋ねる: 「プラグインをどこに作成しますか？」
   - 選択肢を提示する: カレントディレクトリ、../new-plugin-name、カスタムパス
3. bash を使ってディレクトリ構造を作成する:
   ```bash
   mkdir -p plugin-name/.claude-plugin
   mkdir -p plugin-name/skills     # if needed
   mkdir -p plugin-name/commands   # if needed
   mkdir -p plugin-name/agents     # if needed
   mkdir -p plugin-name/hooks      # if needed
   ```
4. Write ツールを使って plugin.json マニフェストを作成する:
   ```json
   {
     "name": "plugin-name",
     "version": "0.1.0",
     "description": "[brief description]",
     "author": {
       "name": "[author from user or default]",
       "email": "[email or default]"
     }
   }
   ```
5. README.md テンプレートを作成する
6. 必要に応じて .gitignore を作成する（.claude/*.local.md など用）
7. 新しいディレクトリを作成する場合は git リポジトリを初期化する

**出力**: プラグインのディレクトリ構造が作成され、コンポーネントの準備完了

## フェーズ 5: コンポーネント実装

**目標**: ベストプラクティスに従って各コンポーネントを作成する

**各コンポーネント種類の実装前に関連スキルをロードすること**:
- Skills: skill-creator スキルをロード
- Commands: command-development スキルをロード
- Agents: agent-development スキルをロード
- Hooks: hook-development スキルをロード
- MCP: mcp-integration スキルをロード
- Settings: plugin-settings スキルをロード

**各コンポーネントに対するアクション**:

### Skills の場合:
1. Skill ツールを使って skill-creator スキルをロードする
2. 各スキルに対して:
   - ユーザーに具体的な使用例を聞く（またはフェーズ3のものを使用）
   - リソースを計画する（scripts/、references/、examples/）
   - スキルのディレクトリ構造を作成する
   - SKILL.md を作成する:
     - 具体的なトリガーフレーズを含む三人称の説明
     - 命令形で書かれた軽量な本文（1,500〜2,000語）
     - サポートファイルへの参照
   - 詳細コンテンツ用のリファレンスファイルを作成する
   - 動作するコード用のサンプルファイルを作成する
   - 必要に応じてユーティリティスクリプトを作成する
3. skill-reviewer エージェントを使って各スキルを検証する

### Commands の場合:
1. Skill ツールを使って command-development スキルをロードする
2. 各コマンドに対して:
   - フロントマター付きのコマンド Markdown を作成する
   - 明確な description と argument-hint を含める
   - allowed-tools を指定する（必要最小限）
   - Claude 向けの指示を書く（ユーザー向けではなく）
   - 使用例とヒントを提供する
   - 該当する場合は関連スキルを参照する

### Agents の場合:
1. Skill ツールを使って agent-development スキルをロードする
2. 各エージェントに対して、agent-creator エージェントを使用する:
   - エージェントの役割の説明を提供する
   - agent-creator が生成する: identifier、whenToUse（例付き）、systemPrompt
   - フロントマターとシステムプロンプト付きのエージェント Markdown ファイルを作成する
   - 適切な model、color、tools を追加する
   - validate-agent.sh スクリプトで検証する

### Hooks の場合:
1. Skill ツールを使って hook-development スキルをロードする
2. 各フックに対して:
   - フック設定を含む hooks/hooks.json を作成する
   - 複雑なロジックにはプロンプトベースのフックを推奨する
   - ポータビリティのために ${CLAUDE_PLUGIN_ROOT} を使用する
   - 必要に応じてフックスクリプトを作成する（scripts/ ではなく examples/ に）
   - validate-hook-schema.sh と test-hook.sh ユーティリティでテストする

### MCP の場合:
1. Skill ツールを使って mcp-integration スキルをロードする
2. 以下を含む .mcp.json 設定を作成する:
   - サーバータイプ（ローカルは stdio、ホスト型は SSE）
   - コマンドと引数（${CLAUDE_PLUGIN_ROOT} 付き）
   - LSP の場合は extensionToLanguage マッピング
   - 必要に応じて環境変数
3. 必要な環境変数を README に記載する
4. セットアップ手順を提供する

### Settings の場合:
1. Skill ツールを使って plugin-settings スキルをロードする
2. README に設定テンプレートを作成する
3. サンプルの .claude/plugin-name.local.md ファイルを作成する（ドキュメントとして）
4. 必要に応じてフック/コマンドに設定読み込みを実装する
5. .gitignore に追加する: `.claude/*.local.md`

**進捗追跡**: 各コンポーネントの完了時に todo を更新する

**出力**: すべてのプラグインコンポーネントが実装済み

## フェーズ 6: 検証と品質チェック

**目標**: プラグインが品質基準を満たし、正しく動作することを確認する

**アクション**:
1. **plugin-validator エージェントを実行する**:
   - plugin-validator エージェントを使ってプラグインを包括的に検証する
   - チェック内容: マニフェスト、構造、命名、コンポーネント、セキュリティ
   - 検証レポートをレビューする

2. **致命的な問題を修正する**:
   - 検証からの致命的なエラーに対処する
   - 実際の問題を示す警告を修正する

3. **skill-reviewer でレビューする**（プラグインにスキルがある場合）:
   - 各スキルに対して skill-reviewer エージェントを使用する
   - 説明の品質、段階的な情報開示、文体をチェックする
   - 推奨事項を適用する

4. **エージェントのトリガーをテストする**（プラグインにエージェントがある場合）:
   - 各エージェントの <example> ブロックが明確か検証する
   - トリガー条件が具体的かチェックする
   - エージェントファイルに対して validate-agent.sh を実行する

5. **フック設定をテストする**（プラグインにフックがある場合）:
   - hooks/hooks.json に対して validate-hook-schema.sh を実行する
   - test-hook.sh でフックスクリプトをテストする
   - ${CLAUDE_PLUGIN_ROOT} の使用を検証する

6. **結果を提示する**:
   - 検証結果の要約
   - 残存する問題
   - 全体的な品質評価

7. **ユーザーに確認する**: 「検証完了。発見された問題: [致命的 N 件]、[警告 N 件]。今すぐ修正しますか、それともテストに進みますか？」

**出力**: プラグインが検証済みでテストの準備完了

---

## フェーズ 7: テストと検証

**目標**: プラグインが Claude Code で正しく動作することをテストする

**アクション**:
1. **インストール手順**:
   - ローカルテストの方法をユーザーに示す:
     ```bash
     cc --plugin-dir /path/to/plugin-name
     ```
   - またはプロジェクトテスト用に `.claude-plugin/` にコピーする

2. **ユーザーが実施する検証チェックリスト**:
   - [ ] トリガー時にスキルがロードされる（トリガーフレーズで質問する）
   - [ ] コマンドが `/help` に表示され、正しく実行される
   - [ ] 適切なシナリオでエージェントがトリガーされる
   - [ ] フックがイベントで起動する（該当する場合）
   - [ ] MCP サーバーが接続する（該当する場合）
   - [ ] 設定ファイルが機能する（該当する場合）

3. **テストの推奨事項**:
   - スキルの場合: 説明のトリガーフレーズを使って質問する
   - コマンドの場合: 様々な引数で `/plugin-name:command-name` を実行する
   - エージェントの場合: エージェントのサンプルに一致するシナリオを作成する
   - フックの場合: `claude --debug` を使ってフックの実行を確認する
   - MCP の場合: `/mcp` を使ってサーバーとツールを検証する

4. **ユーザーに確認する**: 「テスト用のプラグインを準備しました。各コンポーネントのテストをガイドしましょうか、それとも自分でテストしますか？」

5. **ユーザーがガイドを希望する場合**、具体的なテストケースで各コンポーネントのテストを案内する

**出力**: プラグインがテスト済みで動作確認完了

---

## フェーズ 8: ドキュメントと次のステップ

**目標**: プラグインが十分にドキュメント化され、配布の準備ができていることを確認する

**アクション**:
1. **README の完全性を確認する**:
   - README に以下が含まれているかチェックする: 概要、機能、インストール、前提条件、使い方
   - MCP プラグインの場合: 必要な環境変数を記載する
   - フックプラグインの場合: フックの起動を説明する
   - 設定がある場合: 設定テンプレートを提供する

2. **マーケットプレイスエントリを追加する**（公開する場合）:
   - marketplace.json への追加方法をユーザーに示す
   - マーケットプレイスの説明文の作成を支援する
   - カテゴリとタグを提案する

3. **サマリーを作成する**:
   - すべての todo を完了にマークする
   - 作成したものを一覧にする:
     - プラグイン名と目的
     - 作成したコンポーネント（X スキル、Y コマンド、Z エージェントなど）
     - 主要なファイルとその目的
     - 合計ファイル数と構造
   - 次のステップ:
     - テストの推奨事項
     - マーケットプレイスへの公開（希望する場合）
     - 使用に基づくイテレーション

4. **改善を提案する**（オプション）:
   - プラグインを強化できる追加コンポーネント
   - 統合の機会
   - テスト戦略

**出力**: 完全でドキュメント化されたプラグインが使用または公開の準備完了

## 重要な注意事項

### すべてのフェーズを通じて

- **TodoWrite を使用する**: すべてのフェーズで進捗を追跡する
- **Skill ツールでスキルをロードする**: 特定のコンポーネント種類で作業する場合
- **専門エージェントを使用する**（agent-creator、plugin-validator、skill-reviewer）
- **重要な決定ポイントでユーザーの確認を得る**
- **plugin-dev 自体のパターンに従う**: 参考例として
- **ベストプラクティスを適用する**:
  - スキルの三人称説明
  - スキル本文の命令形
  - Claude 向けに書かれたコマンド
  - 強力なトリガーフレーズ
  - ポータビリティのための ${CLAUDE_PLUGIN_ROOT}
  - 段階的な情報開示
  - セキュリティファースト（HTTPS、ハードコードされた認証情報なし）

### 主要な決定ポイント（ユーザーの回答を待つ）

1. フェーズ 1 の後: プラグインの目的を確認
2. フェーズ 2 の後: コンポーネント計画を承認
3. フェーズ 3 の後: 実装に進む
4. フェーズ 6 の後: 問題を修正するか進める
5. フェーズ 7 の後: ドキュメントに進む

### フェーズ別にロードするスキル

- **フェーズ 2**: plugin-structure
- **フェーズ 5**: skill-creator、command-development、agent-development、hook-development、mcp-integration、plugin-settings（必要に応じて）
- **フェーズ 6**: （エージェントが自動的にスキルを使用する）

### 品質基準

すべてのコンポーネントは以下の基準を満たすこと:
- plugin-dev の実証済みパターンに従っている
- 正しい命名規則を使用している
- 強力なトリガー条件がある（スキル/エージェント）
- 動作するサンプルが含まれている
- 適切にドキュメント化されている
- ユーティリティで検証済み
- Claude Code でテスト済み

## ワークフロー例

### ユーザーリクエスト
「データベースマイグレーション管理用のプラグインを作成して」

### フェーズ 1: 調査
- 理解: マイグレーション管理、データベーススキーマのバージョニング
- 確認: ユーザーはマイグレーションの作成、実行、ロールバックを望んでいる

### フェーズ 2: コンポーネント計画
- Skills: 1（マイグレーションのベストプラクティス）
- Commands: 3（create-migration、run-migrations、rollback）
- Agents: 1（migration-validator）
- MCP: 1（データベース接続）

### フェーズ 3: 明確化の質問
- どのデータベースか？（PostgreSQL、MySQL など）
- マイグレーションファイルの形式は？（SQL、コードベース？）
- エージェントは適用前に検証すべきか？
- どの MCP ツールが必要か？（query、execute、schema）

### フェーズ 4-8: 実装、検証、テスト、ドキュメント

---

**フェーズ 1: 調査から開始する**
