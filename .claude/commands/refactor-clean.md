デッドコード、未使用のエクスポート、冗長な実装を安全に削除します。refactor-cleaner エージェントに委譲します。

## 検出ツール

```bash
npx knip              # 未使用のファイルとエクスポート（JS/TS）
npx depcheck          # 未使用の npm 依存関係
npx ts-prune          # TypeScript 未使用エクスポート
npx eslint --report-unused-disable-directives .  # 未使用のディレクティブ

# 言語別
vulture .             # Python デッドコード
deadcode ./...        # Go デッドコード
cargo-udeps           # Rust 未使用依存関係
```

## リスク分類

| リスク | 例 | アクション |
|--------|-----|------------|
| **安全** | 未使用のユーティリティ、到達不能コード | 確信を持って削除 |
| **注意** | コンポーネント、ルート | 動的な使用がないか確認 |
| **危険** | 設定ファイル、エントリーポイント | 徹底的に調査 |

## プロセス

1. **まずテストを実行** — ベースラインを確立
2. **1つだけ削除** — 可能な限り最小の変更
3. **テストを再実行** — 何も壊れていないか確認
4. **何か失敗したら即座にリバート**
5. **次の項目に対して繰り返す**

## 注意項目の削除前

動的な使用をチェック:
```bash
grep -r "dynamicImport\|require(" src/
grep -r "symbolName" --include="*.ts" .
```

## 統合

クリーンアップ後、ほぼ重複している関数を特定:
- 単一の共有実装に統合
- すべての呼び出し元を統一バージョンに更新
- オリジナルを削除

## 最終レポート

```
### クリーンアップ概要
- 削除: [N 行、M ファイル]
- 削除した未使用依存関係: [パッケージ名]
- 統合: [N 件の重複関数をマージ]
- スキップ: [削除が不確実な項目]
- テスト: [X 件成功] — リグレッションなし
```

**ルール:** テストを実行せずに削除しないこと。一度に1つの変更のみ。
