## Hook 候補

| 検出パターン | 防止方法 | Hook 種別 | 実装案 |
|-------------|---------|-----------|--------|
| `.js` ファイルに `console.log(` が残ったままコミット | コミット前に `console.log` を検出して警告 | `PreToolUse(Bash)` — git commit 時 | `grep -rn "console\.log(" src/public/js/ --include="*.js"` でヒット数を出力 |
| `docs/implement/plan.md` を Write した直後に複数回 Edit | plan 生成後すぐに修正している＝生成品質が低い | `PostToolUse(Write)` — plan.md 生成後 | 完了条件・影響範囲・テスト方針の3セクションが揃っているか検査し、不足なら警告 |

## Skill 候補

| 繰り返しパターン | 自動化方法 | Skill 名 | トリガー条件 |
|----------------|-----------|----------|-------------|
| 複数ファイルにまたがる同種文字列の一括置換（「Playwright」→「antigravity」を6+6回）| `grep` で対象箇所を列挙 → 一括 `sed` / Edit | `bulk-replace` | 「〇〇を××に全部置換して」という指示 |
| `console.log` デバッグ追加 → 後でまとめて削除 | デバッグログを専用フラグ変数（`DEBUG_MODE`）で管理し、削除不要にする | `debug-log-cleanup` | `.js` ファイルへの `console.log` 追加を検知 |

## 学習事項

| 観察 | 原因推定 | 改善アクション |
|------|---------|---------------|
| `evidence-based-fix.md` を6回 Edit した後に全体 Write で再作成 | 置換すべき文字列パターンが多く、Edit を繰り返すよりファイル全体を書き直す方が速かったが、最初からそう判断できなかった | 同一ファイルへの Edit が3回を超えたら Write（全体再作成）に切り替えるルールを追加 |
| `browser-verification.md` を異なるパスに2回 Write | 配置先ディレクトリを決める前に作成を始めた | ファイル作成前に `find_file` でパス確認する習慣 |
| `table_ui.js` ↔ `datatable.js` は常にペアで編集 | initComplete コールバックの結合度が高い | 変更時に両ファイルを同時に Read し、セットで計画する |
| JS コメントの言語を直後に変更 | プロジェクトルールで JS コメントの言語が未定義 | `project-conventions.md` に「JSコメントは英語」を明記（PHPとの一貫性は別記） |
