## 学習事項

| 観察 | 原因推定 | 改善アクション |
|------|---------|---------------|
| `excel_range_select_copy.js` を3アプローチで修正。各試行後にブラウザで確認し、失敗したら全書き直し | CSS の重ね順（`z-index`）と `input` の `pointer-events` の理解不足。`box-shadow` が `input` に隠れることを最初に把握していれば1回で済んだ | 修正前に「セル内要素の重ね順」を確認するチェック項目を追加。`verify-before-fix` スキルの活用を強化 |
| `proposal.md` の `Write` が同日に7回超発生（要件が実装中に変化し続けた） | 実装を始めてから要件が変わるたびに proposal を書き直す「研究と実装の混在」が発生 | 要件確定まで実装に入らない。`CLAUDE.md` の「研究と実装分離」ルールを proposal に明記 |
| `settings.json` の PreToolUse matcher を変更→即 revert（6分） | フックの実際の動作をローカルでテストできないまま変更 | `settings.json` 変更時は変更内容を Slack や logs で確認してから次の変更を加える。変更1件ずつ検証 |
| docker-compose.yml の PMA_HOSTS 変更→revert（47分） | phpMyAdmin の `PMA_HOSTS` 環境変数の動作を検証せずに実装 | Docker 設定変更は「コンテナ再起動して確認」を完了条件に含める |
| Slackフック4ファイルへの同一パターン追加が手動で行われた | 共通ロジックが4ファイルに分散しており、1か所修正しても他に伝播しない | `is_work_hours()` のような横断的条件は `lib/` に集約済みなので `OK`。ただし「同一パターンを複数ファイルに追加」は自動化の候補として記録 |
| `order.blade.php` のパネル配置が3回変更された | 「コンポーネント内に配置」vs「ホーム画面のレイアウトで配置」の設計判断が着手前に固まっていなかった | UI コンポーネントの配置はスケッチまたは proposal で「どのDOM階層に置くか」を事前定義する |
