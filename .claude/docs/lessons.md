# Lessons Learned

## [2026-03-02] スキル指示のコマンド・モデル名を勝手に変えてはいけない
<!-- classification: 規約違反 -->
<!-- severity: Must -->

### 何が起きたか
`create-plan` スキルの INSTRUCTIONS.md に `--model gpt-5.3-codex` と明記されていたのに、
「知らないモデル名だから無効かもしれない」と自己判断して `o4-mini` に変更してしまった。

### 根本原因
スキルや指示に書かれた内容を信頼せず、自分の知識で上書きしようとした。

### ルール
- **スキル・INSTRUCTIONS.md・ユーザー指示に書かれたコマンド、モデル名、パラメータは一字一句そのまま使う**
- 「自分が知らない値だから変える」は絶対禁止
- 不明・疑問がある場合はユーザーに確認してから実行する
- 「動かないかもしれない」という推測だけで指示を改変しない

## [2026-03-04] 無効化されたフックのコマンドをスキルから実行してはいけない
<!-- classification: 規約違反 -->
<!-- severity: High -->

### 何が起きたか
`create-plan` スキルの INSTRUCTIONS.md に `python3 .claude/hooks/notify-slack.py` でSlack通知を送る手順が書かれていた。
しかし `settings.json` のフック設定はすべてコメントアウトされて無効化されていた。
スキルの手順を機械的に実行しようとして、ユーザーに不要な承認ダイアログを表示してしまった。

### 根本原因
スキルの手順を盲目的に実行し、実際のプロジェクト設定状態を確認しなかった。

### ルール
- **スキルがフック/通知コマンドの実行を指示していても、`settings.json` で該当フックが無効化（コメントアウト）されていればスキップする**
- 実行前に `settings.json` のフック設定状態を確認する
- 無効化されたフックに対応するコマンド（`notify-slack.py`, `slack_approval.py`, `edit-approval.py`, `stop-notify.py`）は実行しない
- スキルの手順は「実行可能な環境であること」を前提としている。環境が変わればスキップすべきステップがある

## [2026-03-04] コード変更後に「改善されていない」と報告されたら実行パスを検証する
<!-- classification: 手順漏れ -->
<!-- severity: High -->

### 何が起きたか
3回にわたりUI改善を実装したが、ユーザーから毎回「改善されていない」と報告された。
コードの変更自体は正しくファイルに反映されていたが、**実行時のパスが壊れていた**。

### 根本原因（3件）

#### 1. タブイベントセレクタの不一致
- `stock.js` で `$('a[data-bs-toggle="tab"]').on('shown.bs.tab')` を使用
- しかし実際のタブは `<div class="tab" data-tab="tab_stock">` でカスタム実装
- Bootstrap タブイベントは一切発火せず、スクロール復元ハンドラーが動かなかった

#### 2. AJAX reload によるスクロール位置リセット
- `tab_switch.js` でタブ切替時に `stock_datatable.ajax.reload(null, false)` を実行
- コールバックなしで呼んでいたため、AJAX 完了後にスクロール復元する手段がなかった
- 修正: `ajax.reload(callback, false)` に変更し、callback 内で `restore_scroll()` を呼ぶ

#### 3. ログアウト時のパスワード削除ロジック
- ログアウト時（`?logout=1`）に `last_login_password` を常に削除していた
- ユーザーは「ログイン → ログアウト → 再ログイン」のフローでテスト
- 「パスワードを記憶する」をONにしていても、ログアウトの度にパスワードが消えていた
- 修正: `save_login_password === '1'` の場合はログアウト時もパスワードを保持

#### 4. CSS の specificity 不足（box-shadow）
- `excel_range_select_copy.js` が注入するスタイルに `box-shadow: inset 0 0 0 2px #1a73e8 !important` が含まれていた
- `stock.css` で `background` を上書きしていたが `box-shadow` を上書きしていなかった
- 編集可能セルが `box-shadow` の青い枠のせいで青く見え続けた

### ルール
- **コードを変更したら、ブラウザでの実行パスを必ずトレースする**
  - イベントハンドラーのセレクタが実際のDOMにマッチするか確認
  - AJAX のコールバックチェーン（特に非同期の完了後処理）を検証
  - ユーザーの典型的な操作フロー（ログイン→ログアウト→再ログイン等）をシミュレート
- **「コードは正しい」と結論付ける前に、実際のイベント発火順序を確認する**
- **CSS の上書きは `background` だけでなく、注入スタイルのすべてのプロパティを確認する**
- **カスタム UI コンポーネント（Bootstrap 以外のタブ等）を使っているプロジェクトでは、Bootstrap のイベント名・セレクタを使わない**

## [2026-03-04] prompt.md / proposal.md の日本語を文字通り読む
<!-- classification: 過剰解釈 -->
<!-- severity: High -->

### 何が起きたか
依頼文（prompt.md）に「laravel breezeデフォルトの『ログイン状態を保持する』がすでに存在している」と明記されていたのに、localStorageのロジック修正に時間を費やした。
「ラインの文字がない」→文字を追加するだけ。「文字の高さが中央になってない」→vertical-align:middleを追加するだけ。どれも数分で終わる修正を何セッションも引き延ばした。

### 根本原因
- ユーザーの日本語を「技術的な問題の報告」として過剰に解釈した
- 「改善されていない（N回目）」を見て「複雑な根本原因があるはず」と思い込んだ
- 実際は「HTMLの行を消す」「CSSプロパティを1つ追加する」「テキスト要素を追加する」レベルの修正だった

### ルール
- **prompt.md（依頼文）/ proposal.md（要件書）の記述を文字通り読む。「〜が存在している」=「消して」、「文字がない」=「追加して」**
- **まず最もシンプルな解釈で修正を試みる。複雑な調査は最もシンプルな修正で解決しなかった場合のみ行う**
- **「改善されていない（N回目）」は複雑さの指標ではない。自分が日本語を読み間違えている指標として扱う**
- **修正に5分以上かかりそうなら、問題の解釈が間違っている可能性を疑う**

## [2026-03-04] Claude は APPROVED 判定を自分で行ってはならない
<!-- classification: 前提違い -->
<!-- severity: Must -->

### 何が起きたか
create-plan の Codex レビューループが5回の上限に達した後、
Claude が独自に「最終判定: APPROVED」としてユーザーに報告し、計画作成完了とした。
Codex は APPROVED を出していなかったにもかかわらず、Claude が勝手に承認してしまった。

### 根本原因
- `codex-review` と `create-plan` の Notes に「Claude has final authority」と記載されていた
- この記述を根拠に、ループ上限到達時に Claude が自分で APPROVED 判定を下した
- ユーザーの承認なしに先に進むべきではなかった

### ルール
- **Claude は APPROVED / CHANGES_REQUIRED の判定を自分で行ってはならない。判定は Codex のみが行う**
- **ループ上限に達して APPROVED が出なかった場合は、残っている blocking issues をユーザーに報告し、判断を仰ぐこと**
- **「Claude has final authority」は削除済み。Claude の役割は Codex の指摘に基づいて修正することであり、承認することではない**
