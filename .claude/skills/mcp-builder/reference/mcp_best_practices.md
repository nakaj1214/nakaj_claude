# MCP サーバーベストプラクティス

## クイックリファレンス

### サーバーの命名
- **Python**: `{service}_mcp` (例: `slack_mcp`)
- **Node/TypeScript**: `{service}-mcp-server` (例: `slack-mcp-server`)

### ツールの命名
- snake_case にサービスプレフィックスを付ける
- フォーマット: `{service}_{action}_{resource}`
- 例: `slack_send_message`, `github_create_issue`

### レスポンスフォーマット
- JSON と Markdown の両方のフォーマットをサポート
- JSON はプログラム処理用
- Markdown は人間の可読性用

### ページネーション
- 常に `limit` パラメータを尊重する
- `has_more`, `next_offset`, `total_count` を返す
- デフォルトは 20〜50 アイテム

### トランスポート
- **Streamable HTTP**: リモートサーバー、マルチクライアントシナリオ用
- **stdio**: ローカル統合、コマンドラインツール用
- SSE は避ける（Streamable HTTP に非推奨化）

---

## サーバー命名規則

以下の標準化された命名パターンに従います:

**Python**: `{service}_mcp` フォーマット（アンダースコア付き小文字）
- 例: `slack_mcp`, `github_mcp`, `jira_mcp`

**Node/TypeScript**: `{service}-mcp-server` フォーマット（ハイフン付き小文字）
- 例: `slack-mcp-server`, `github-mcp-server`, `jira-mcp-server`

名前は、汎用的で、統合するサービスを説明し、タスクの説明から推測しやすく、バージョン番号を含まないものにします。

---

## ツールの命名とデザイン

### ツールの命名

1. **snake_case を使用**: `search_users`, `create_project`, `get_channel_info`
2. **サービスプレフィックスを含める**: MCP サーバーが他の MCP サーバーと並行して使用される可能性を想定
   - `send_message` ではなく `slack_send_message` を使用
   - `create_issue` ではなく `github_create_issue` を使用
3. **アクション指向**: 動詞で始める（get, list, search, create 等）
4. **具体的に**: 他のサーバーと衝突する可能性のある汎用名を避ける

### ツールデザイン

- ツールの説明は機能を狭く曖昧さなく記述しなければならない
- 説明は実際の機能と正確に一致しなければならない
- ツールアノテーションを提供する（readOnlyHint, destructiveHint, idempotentHint, openWorldHint）
- ツールの操作は集中的かつアトミックに保つ

---

## レスポンスフォーマット

データを返すすべてのツールは複数のフォーマットをサポートすべきです:

### JSON フォーマット (`response_format="json"`)
- 機械可読な構造化データ
- 利用可能なすべてのフィールドとメタデータを含む
- 一貫したフィールド名と型
- プログラム処理に使用

### Markdown フォーマット (`response_format="markdown"`, 通常デフォルト)
- 人間が読みやすいフォーマット済みテキスト
- 見出し、リスト、フォーマットを使用して明確に
- タイムスタンプを人間が読めるフォーマットに変換
- ID を括弧内に添えて表示名を表示
- 冗長なメタデータを省略

---

## ページネーション

リソースを一覧表示するツールの場合:

- **常に `limit` パラメータを尊重する**
- **ページネーションを実装**: `offset` またはカーソルベースのページネーションを使用
- **ページネーションメタデータを返す**: `has_more`, `next_offset`/`next_cursor`, `total_count` を含む
- **すべての結果をメモリにロードしない**: 特に大規模データセットで重要
- **合理的なデフォルト制限**: 20〜50 アイテムが一般的

ページネーションレスポンスの例:
```json
{
  "total": 150,
  "count": 20,
  "offset": 0,
  "items": [...],
  "has_more": true,
  "next_offset": 20
}
```

## トランスポートオプション

### Streamable HTTP

**最適な用途**: リモートサーバー、Web サービス、マルチクライアントシナリオ

**特徴**:
- HTTP 上の双方向通信
- 複数の同時クライアントをサポート
- Web サービスとしてデプロイ可能
- サーバーからクライアントへの通知が可能

**使用する場面**:
- 複数のクライアントに同時サービス提供
- クラウドサービスとしてデプロイ
- Web アプリケーションとの統合

### stdio

**最適な用途**: ローカル統合、コマンドラインツール

**特徴**:
- 標準入出力ストリーム通信
- シンプルなセットアップ、ネットワーク設定不要
- クライアントのサブプロセスとして実行

**使用する場面**:
- ローカル開発環境用のツール構築
- デスクトップアプリケーションとの統合
- シングルユーザー、シングルセッションのシナリオ

**注意**: stdio サーバーは stdout にログを出力してはならない（ログには stderr を使用）

### トランスポートの選択

| 基準 | stdio | Streamable HTTP |
|-----------|-------|-----------------|
| **デプロイ** | ローカル | リモート |
| **クライアント** | 単一 | 複数 |
| **複雑さ** | 低 | 中 |
| **リアルタイム** | 不可 | 可 |

## セキュリティベストプラクティス

### 認証と認可

**OAuth 2.1**:
- 認定機関の証明書を使用した安全な OAuth 2.1 を使用
- リクエスト処理前にアクセストークンを検証
- 自分のサーバー専用に意図されたトークンのみを受け入れる

**API キー**:
- API キーは環境変数に保存し、コードに書かない
- サーバー起動時にキーを検証
- 認証失敗時に明確なエラーメッセージを提供

### 入力バリデーション

- ディレクトリトラバーサルを防ぐためにファイルパスをサニタイズ
- URL と外部識別子を検証
- パラメータのサイズと範囲をチェック
- システムコールでのコマンドインジェクションを防止
- すべての入力にスキーマバリデーション（Pydantic/Zod）を使用

### エラーハンドリング

- 内部エラーをクライアントに公開しない
- セキュリティ関連のエラーをサーバー側でログ記録
- 有用だが情報を明かさないエラーメッセージを提供
- エラー後にリソースをクリーンアップ

### DNS リバインディング保護

ローカルで実行する Streamable HTTP サーバーの場合:
- DNS リバインディング保護を有効にする
- すべての着信接続で `Origin` ヘッダーを検証
- `0.0.0.0` ではなく `127.0.0.1` にバインド

## ツールアノテーション

クライアントがツールの動作を理解できるようアノテーションを提供します:

| アノテーション | 型 | デフォルト | 説明 |
|-----------|------|---------|-------------|
| `readOnlyHint` | boolean | false | ツールは環境を変更しない |
| `destructiveHint` | boolean | true | ツールは破壊的な更新を行う可能性がある |
| `idempotentHint` | boolean | false | 同じ引数での繰り返し呼び出しに追加効果がない |
| `openWorldHint` | boolean | true | ツールは外部エンティティと対話する |

**重要**: アノテーションはヒントであり、セキュリティ保証ではありません。クライアントはアノテーションのみに基づいてセキュリティ上重要な判断をすべきではありません。

---

## エラーハンドリング

- 標準的な JSON-RPC エラーコードを使用
- ツールエラーは結果オブジェクト内で報告（プロトコルレベルのエラーではなく）
- 有用で具体的なエラーメッセージと次のステップの提案を提供
- 内部実装の詳細を公開しない
- エラー時にリソースを適切にクリーンアップ

エラーハンドリングの例:
```typescript
try {
  const result = performOperation();
  return { content: [{ type: "text", text: result }] };
} catch (error) {
  return {
    isError: true,
    content: [{
      type: "text",
      text: `Error: ${error.message}. Try using filter='active_only' to reduce results.`
    }]
  };
}
```

## テスト要件

包括的なテストは以下をカバーすべきです:

- **機能テスト**: 有効/無効な入力での正しい実行を検証
- **統合テスト**: 外部システムとの対話をテスト
- **セキュリティテスト**: 認証、入力サニタイゼーション、レートリミティングを検証
- **パフォーマンステスト**: 負荷時の動作、タイムアウトをチェック
- **エラーハンドリング**: 適切なエラー報告とクリーンアップを確認

## ドキュメント要件

- すべてのツールと機能の明確なドキュメントを提供
- 動作する例を含める（主要な機能ごとに少なくとも3つ）
- セキュリティに関する考慮事項をドキュメント化
- 必要な権限とアクセスレベルを明記
- レートリミットとパフォーマンス特性をドキュメント化
